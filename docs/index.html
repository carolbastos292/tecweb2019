2. Requisitos

Os requisitos de um sistema são descrições dos serviços fornecidos pelo sistema e as suas restrições operacionais. Esses requisitos refletem as necessidades dos clientes de um sistema que ajuda a resolver algum problema, por exemplo, controlar um dispositivo, enviar um pedido ou encontrar informações (SOMMERVILLE,2011).

2.1 Não funcionais

São restrições aos serviços ou funções oferecidos pelo sistema. Incluem restrições de timing, restrições no processo de desenvolvimento e restrições im postas pelas normas (SOMMERVILLE,2011).

Para acessar os requisitos não funcionais clique aqui.

2.2 Funcionais

Os requisitos funcionais são declarações que o sistema deve fornecer, de como o sistema deve reagir a entradas específicas e de como o sistema deve se comportar em determinadas situações (SOMMERVILLE,2011).

5.2 Estrutura principal

Existem 4 pastas principais dentro do projeto:
  1. flask - Nesta pasta fica o backend que realiza as requisições ao banco de dados, bem como o tratamento das entradas vindas do telas da aplicação.
  2. nginx - É nesta pasta que as configurações a serem usadas pelo Nginx se encontra.
  3. postgres - Esta é onde fica armazenados qualquer script relacionado diretamente ao banco de dados, como a configuração, modelagem e popularização do banco de dados.
  4. react - Aqui fica a codificação, em React, das telas que serão usadas pela aplicação.

5.2.1 Estruturas internas

5.2.1.1 Flask

Dentro do projeto existem 3 pastas principais:
  1. Angelica Nesta pasta temos os scripts para a criação da API flask:
    • A inicialização do ambiente: __init__.py
    • fix: uma correção de bug;
    • A instanciação do banco: database.py
    • A declaração de como será declarado cada coluna e tabelas disponíveis no banco: models.py
    • A separação de funções gerais a serem usadas: methods.py
    • As API's para realização das funções necessárias do sistema: views.py
    • chore: atualizando: features, gerenciador de pacotes, etc - mudança de código
  2. __Pycache__ Possui os dados gerados durante a execução do sistema.
  3. env Possui as configurações do ambiente virtual que é usado para testes.

5.2.1.2 NGINX

Nesta pasta existe apenas o arquivo "nginx.conf", que serve para configurar o Nginx para ser usado.

5.2.1.3 Postgres

Aqui existem dois arquivos principais:
  1. Adatabase.conf : Este arquivo armazena as configurações necessárias para o banco, como o nome do modelo a ser usado, usuário padrão, senha e entre outros.
  2. init.sql : Este é o arquivo responsável de modelar e popular o banco, caso o mesmo não esteja ainda instanciado.

5.2.1.4 React

Este é o local onde a aplicação React se encontrará, onde em sua raiz existe um arquivo importante e uma pasta a ser melhor explicada:
  1. package.json: Arquivo que possui as dependências e versões necessárias para que a aplicação funcione normalmente.
  2. src: Nesta pasta que se encontra os arquivos .js que formam a aplicação como um todo.
A pasta src é dividida em quatro pastas principais:
  1. assets: Onde serão colocados arquivos como imagens, sons, fontes e entre outros.
  2. componentes: Components React que serão usados no projeto.
  3. layouts: Que contem os layouts gerais do sistema.
  4. screens: Onde se encontram os arquivos responsáveis em juntar os componentes e criar a tela a ser exibida.

5.2.2 Por que essa distribuição?

Essa distribuição fi realizada com o intuito de otimizar e organizar cada parte do sistema de forma a facilitar o uso do Docker-Compose, que configura cada uma das pastas como um container docker separado um do outro, permitindo que uma parte do sistema, caso seja necessário atualizações ou manutenções, possa ser alterada em tempo de execução e não afetar as demais.

5.3 Boas Práticas

5.3.1 Git

5.3.1.1 Estrutura de Mensagem

Uma mensagem de commit consiste em três partes distintas separadas por uma linha em branco: título, um corpo (opcional) e um rodapé (opcional). O layout é assim:

O título consiste no tipo de mensagem e assunto

5.3.1.2 Tipo de Mensagem (Type)

O tipo está contido no título e pode ser um desses tipos:
  • feat: uma nova feature;
  • fix: uma correção de bug;
  • docs: mudanças na documentação;
  • style: formatação, falta de dois pontos, etc - sem alteração de código;
  • refactor: refatorar código de produção;
  • test: adicionar testes, teste de refatoração - nenhuma mudança de código de produção;
  • chore: atualizando: features, gerenciador de pacotes, etc - mudança de código

5.3.1.3 Assunto da Mensagem (Subject)

  • Os assuntos não devem ter mais de 50 caracteres, devem começar com letra maiúscula;
  • Use um tom imperativo para descrever o que um commit faz, e não o que ele fez;
  • Por exemplo, utilize Criar, não criou ou criado.

5.3.1.4 Rodapé da Mensagem (Footer)

O rodapé é opcional e é utilizado para referenciar códigos para rastrear problemas.

5.3.2 Back-end

Ao escrever os códigos para um sistema, o desenvolvedor deve ter em mente que seus scripts devem estar legíveis tanto para si quanto para outros que possam ler e contribuir no desenvolvimento do projeto. Além disso, ao fazer uso das boas práticas de programação, seus códigos possuem maiores chances de produzir menos erros. Caso ocorra algum erro de execução, será mais fácil de identificá-lo.

Python - Layout de código

  • Indentação: identação é obrigatória. Utilizar 4 espaços por nível de indentação;
  • Limite máximo por linha: 79 caracteres;
  • Comentários e Documentação: textos curtos e relevantes no código, informando a finalidade e o retorno dos métodos;
  • Convenções de nomes: nomes adequados e padronizados, indicando, por exemplo, o que uma variável irá armazenar ou o que um determinado método irá fazer;
  • Convenções de nomes: nomes adequados e padronizados, indicando, por exemplo, o que uma variável irá armazenar ou o que um determinado método irá fazer;
  • Tratamento de erros (try/catch): fazer tratamento de erros, mostrando uma mensagem quando uma exceção ocorrer;
  • Padrões de Projetos (Design Patterns): MVC (Model-View-Controller;
  • Testar e Depurar: depurar para analisar o código em tempo de execução e manter a realização de testes;
  • Tamanho: nomes de classes curtos;
  • Snake Case: é a prática de escrever palavras ou frases compostas nas quais os elementos são separados por um caractere sublinhado (_) e nenhum espaço .
